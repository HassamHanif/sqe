Section 1 — What is Software Testing?

Slide 2–3
IEEE Definition

"The process of evaluating a software system to detect differences between expected and actual behaviour, and to assess whether the software meets specified requirements."

Three key ideas:

  • Evaluation of the entire product (not just parts)
  • Detection of discrepancies between expected and actual behaviour
  • Assessment against functional and non-functional requirements

Four Purposes of Testing

PurposeDescriptionExample
Defect DetectionFind bugs before software reaches usersLogin form accepts blank passwords — caught in integration testing
Quality AssuranceProvide confidence system meets quality standardsAll regression tests pass before a production release
Risk MitigationIdentify high-risk areas earlyPayment processing tested under load before launch
Requirements VerificationConfirm software behaves as specifiedEach acceptance criterion in SRS has a matching test case

Testing vs. Debugging — Key Distinction

AspectTestingDebugging
GoalFind defects — show the software has failuresFix defects — locate and remove root cause
Who does itTesters (and developers in unit testing)Developers primarily
WhenThroughout SDLC — planned and systematicAfter a defect is reported or found
OutputTest results: pass/fail, defect reportsFixed code + verification the fix worked
NatureDestructive — tries to break the systemConstructive — repairs the system
AutomationHighly automatableManual analysis; some tool assistance
Real-World Example — Ariane 5 Rocket (1996)
64-bit float converted to 16-bit integer without a guard = Error. The overflow bug in source code = Fault. Rocket self-destructing 37 seconds after launch = Failure. Cost: ~$370 million. The specific trajectory data combination was never tested.

Section 2 — Verification vs. Validation (V&V)

Slide 4–5

Verification

"Are we building the product right?"

  • Checks against specifications and design
  • Static process — no code execution
  • Techniques: reviews, walkthroughs, inspections, static analysis
  • Example: Does the design document match the SRS?

Validation

"Are we building the right product?"

  • Checks against user needs and business goals
  • Dynamic process — involves executing the software
  • Techniques: functional testing, UAT, system testing
  • Example: Does the system do what the user actually needs?

V&V Across the SDLC

SDLC PhaseVerification (Static)Validation (Dynamic)
RequirementsReview docs for completeness, consistencyPrototyping, user interviews
DesignDesign reviews, walkthroughsUI mockup reviews with users
ImplementationCode reviews, static analysisUnit testing
IntegrationIntegration reviews — check interfacesIntegration testing
System TestingCheck test plans against RTMSystem testing against all requirements
AcceptanceAudit that all planned tests were runUAT — satisfies customer business needs?
Real-World Example — NASA Mars Climate Orbiter (1999)
$327.6M orbiter lost because one team used metric units (Newton-seconds) and another used imperial (pound-force seconds). Verification passed — each module worked correctly per its own specification. Validation failed — the combined system did not do what was actually needed. Cost: $327.6 million.

Section 3 — The 7 Principles of Software Testing (ISTQB)

Slide 6–7

Formalised by ISTQB CTFL Syllabus v4.0 (2023). Universal foundations every tester must know.

1️⃣
Testing Shows Presence of Defects
Testing can prove defects exist — it CANNOT prove software is defect-free. Even after extensive testing, undiscovered defects may still exist. (Dijkstra, 1972: "Testing can show the presence of bugs, but never their absence.") Passing all tests does NOT guarantee correctness.
2️⃣
Exhaustive Testing is Impossible
Testing every possible input, path, and precondition is infeasible. A login form with a 12-character password field has 95^12 ≈ 5.4 × 10^23 combinations. Strategy: use risk analysis and equivalence partitioning to prioritise.
3️⃣
Early Testing Saves Money (Shift-Left Testing)
Cost of fixing a defect increases dramatically the later it is found. Requirements: ×1 | Design: ×5 | Coding: ×10 | Unit Testing: ×20 | System Testing: ×40 | Production: ×100+. (Boehm, 1981 — the '1-10-100 rule')
4️⃣
Defect Clustering (Pareto Principle)
~20% of modules contain ~80% of all defects. High-risk modules: new features, complex logic, third-party integrations, legacy code. Use historical defect data to identify 'hot-spot' modules. Other modules still require testing — clustering does not mean they are defect-free.
5️⃣
Pesticide Paradox
If the same tests are run repeatedly, they stop finding new defects. Just as insects develop resistance to pesticides, software becomes 'immune' to the same tests. Coined by Boris Beizer (1990). Solution: regularly review and update test suites; add new test cases; use mutation testing.
6️⃣
Testing is Context-Dependent
No universal testing approach exists. Safety-critical (avionics, medical) → DO-178C, MC/DC, formal verification. E-commerce → functional, performance, security. Mobile app → usability, compatibility, network conditions. Internal CRUD tool → basic happy path and negative tests.
7️⃣
Absence-of-Errors Fallacy
Fixing all known defects does NOT guarantee a successful product. If the system does not meet user needs, it is still a failure. Links to Validation. Example: Microsoft Zune (2006) — worked as specified with no known defects, but failed commercially because users preferred the iPod ecosystem. Defect-free ≠ successful.

Section 4 — Software Testing Lifecycle (STLC)

Slide 8–11
Definition

A systematic process for planning, executing, and closing testing activities. Each phase has Entry Criteria, Activities, and Exit Criteria. Runs in parallel with the SDLC.

Overview — 6 Phases

PhaseKey EntryCore ActivityKey Exit/Output
1. Requirement AnalysisRequirements document availableIdentify testable requirements; prepare RTMRTM ready; test feasibility report
2. Test PlanningRequirements signed offWrite test plan; estimate effort; assign resourcesApproved Test Plan; schedule
3. Test Case DesignTest plan approvedWrite and review test cases; prepare test dataReviewed and approved test cases and data
4. Environment SetupTest cases readyConfigure hardware/software; smoke testReady test environment; smoke test pass
5. Test ExecutionEnvironment and cases readyExecute tests; log defects; re-test fixesAll tests run; defects logged
6. Test ClosureExit criteria met or deadline reachedTest summary report; lessons learned; metricsSigned-off Closure Report; metrics dashboard

STLC Phases — Detailed

1
Requirement Analysis — Identify testable vs. non-testable requirements; build RTM; identify test types needed; automation feasibility.
2
Test Planning — Define test strategy; estimate effort; assign roles; select tools; define entry/exit criteria; perform risk analysis.
3
Test Case Design — Write clear, traceable, independent, repeatable test cases following a standard template.
4
Environment Setup — Mirror production as closely as possible; run smoke test before full execution.
5
Test Execution — Execute tests; log defects with severity/priority; re-test fixes; run regression tests.
6
Test Closure — Test summary report; DRE metric; lessons learned; archive all test artefacts.

Section 7 — Cost of Quality (CoQ)

Slide 14
Definition

CoQ = Cost of Conformance + Cost of Non-Conformance. Introduced by Feigenbaum (1956); popularised by Crosby (1979): "Quality is free. What costs money are the unquality things."

Cost of Conformance (Good)

Prevention Costs: Training, process improvement, requirements reviews, design reviews, standards creation

Appraisal Costs: Testing (all types), inspections, audits, test environment setup, test tool licences

Cost of Non-Conformance (Bad)

Internal Failure: Rework, debugging, re-testing before delivery, schedule delays

External Failure: Production bug fixes, customer support, recalls, legal liability, reputation damage, lost customers

Real-World Cost Examples

IncidentYearCostCategory
Therac-25 radiation overdoses1985-87Lives lost + legalExternal failure
Pentium FDIV bug (Intel recall)1994~$475 millionExternal failure
Y2K remediation1999~$300 billionPrevention cost
Healthcare.gov launch failure2013~$2 billionExternal failure
CrowdStrike global IT outage2024~$10 billion+External failure
💡 Automation Impact: With automated testing, the marginal cost of running tests ≈ $0 once written. Google runs ~150 million automated test cases per day, making conformance (prevention) costs minimal compared to failure costs.

Activity 1

Testing vs. Debugging Identification

Classify each scenario as Testing, Debugging, or Both:

1. A developer runs all unit tests after changing a function.
✅ Answer: Both

Running the unit tests = Testing (systematic execution to detect failures). Changing the function in response to a failure = Debugging. Both activities occur together in TDD-style workflow.

2. A QA engineer writes test cases for a new payment module.
✅ Answer: Testing

Writing test cases is a Testing activity (specifically, the Test Case Design phase of STLC). No code is being fixed — the goal is to plan how to detect defects.

3. A developer uses a step-through debugger to find why a variable is null.
✅ Answer: Debugging

Using a debugger to locate the root cause of a null variable is pure Debugging — constructive activity aimed at finding and fixing the root cause, not detecting failures.

4. A tester finds a crash and logs a defect report.
✅ Answer: Testing

Finding the failure and logging a defect report is Testing — destructive in nature, aimed at detecting that the software has a failure. The actual fixing is left to the developer (Debugging).

🔬 CLO-2 — Understand

4 Explain the difference between Verification and Validation. Why are both required throughout the SDLC? Apply ▼ Show
6 Marks

Verification asks "Are we building the product right?" — it is a static process that checks artefacts (requirements, design, code) against specifications without executing the software.

Validation asks "Are we building the right product?" — it is a dynamic process that executes the software and checks whether it meets actual user needs.

Both are required because: Verification alone can produce a technically correct system that solves the wrong problem. Validation alone without verification misses structural defects in design and code.

Quick Revision — Key Facts

Six Key Takeaways

1
Software Testing (IEEE): Evaluating a system to detect differences between expected and actual behaviour and to assess whether it meets specified requirements.
2
V&V: Verification = "Building it right?" (static, against specs). Validation = "Building the right thing?" (dynamic, against user needs). Both required.
3
7 ISTQB Principles: Shows defects | Exhaustive impossible | Early saves cost | Defect clustering | Pesticide paradox | Context-dependent | Absence-of-errors fallacy
4
STLC (6 phases): Requirement Analysis → Test Planning → Test Case Design → Environment Setup → Test Execution → Test Closure
5
Error → Fault → Failure: Human mistake → flaw in artefact → observable incorrect behaviour
6
Cost of Quality: Conformance (prevention + appraisal) + Non-Conformance (internal + external failure)
