Section 1 — What is Software Testing?

IEEE Definition:

"The process of evaluating a software system to detect differences between expected and actual behaviour, and to assess whether the software meets specified requirements."

Three key ideas: (1) Evaluation of the entire product, (2) Detection of discrepancies between expected and actual behaviour, (3) Assessment against functional and non-functional requirements.

Four Purposes of Testing

PurposeDescriptionExample
Defect DetectionFind bugs before software reaches usersLogin form accepts blank passwords — caught in integration testing
Quality AssuranceProvide confidence system meets quality standardsAll regression tests pass before a production release
Risk MitigationIdentify high-risk areas earlyPayment processing tested under load before launch
Requirements VerificationConfirm software behaves as specifiedEach acceptance criterion in SRS has a matching test case

Testing vs. Debugging — Key Distinction

Real-World Example — Ariane 5 Rocket (1996)

64-bit float converted to 16-bit integer without a guard = Error. The overflow bug in source code = Fault. Rocket self-destructing 37 seconds after launch = Failure. Cost: ~$370 million. The specific trajectory data combination was never tested.

Section 2 — Verification vs. Validation (V&V)

VERIFICATION

"Are we building the product right?"

Checks against specifications and design. Static process — no code execution required.

Techniques: reviews, walkthroughs, inspections, static analysis

Example: Does the design document match the SRS?

VALIDATION

"Are we building the right product?"

Checks against user needs and business goals. Dynamic process — involves executing the software.

Techniques: functional testing, UAT, system testing

Example: Does the system do what the user actually needs?

V&V Across the SDLC

SDLC PhaseVerification (Static)Validation (Dynamic)
RequirementsReview docs for completeness, consistencyPrototyping, user interviews
DesignDesign reviews, walkthroughsUI mockup reviews with users
ImplementationCode reviews, static analysisUnit testing
IntegrationIntegration reviews — check interfacesIntegration testing
System TestingCheck test plans against RTMSystem testing against all requirements
AcceptanceAudit that all planned tests were runUAT — satisfies customer business needs?

Real-World Example — NASA Mars Climate Orbiter (1999)

$327.6M orbiter lost because one team used metric units (Newton-seconds) and another used imperial (pound-force seconds). Verification passed — each module worked correctly per its own specification. Validation failed — the combined system did not do what was actually needed. Cost: $327.6 million.

Section 3 — The 7 Principles of Software Testing (ISTQB)

Formalised by ISTQB CTFL Syllabus v4.0 (2023). Universal foundations every tester must know.

1
Testing Shows Presence of Defects
Testing can prove defects exist — it CANNOT prove software is defect-free. Even after extensive testing, undiscovered defects may still exist. (Dijkstra, 1972: "Testing can show the presence of bugs, but never their absence.") Passing all tests does NOT guarantee correctness.
2
Exhaustive Testing is Impossible
Testing every possible input, path, and precondition is infeasible. A login form with a 12-character password field has 95^12 ≈ 5.4 × 10^23 combinations. Strategy: use risk analysis and equivalence partitioning to prioritise.
3
Early Testing Saves Money (Shift-Left Testing)
Cost of fixing a defect increases dramatically the later it is found. Requirements: ×1 | Design: ×5 | Coding: ×10 | Unit Testing: ×20 | System Testing: ×40 | Production: ×100+. (Boehm, 1981 — the '1-10-100 rule')
4
Defect Clustering (Pareto Principle)
~20% of modules contain ~80% of all defects. High-risk modules: new features, complex logic, third-party integrations, legacy code. Use historical defect data to identify 'hot-spot' modules. Other modules still require testing — clustering does not mean they are defect-free.
5
Pesticide Paradox
If the same tests are run repeatedly, they stop finding new defects. Just as insects develop resistance to pesticides, software becomes 'immune' to the same tests. Coined by Boris Beizer (1990). Solution: regularly review and update test suites; add new test cases; use mutation testing.
6
Testing is Context-Dependent
No universal testing approach exists. Safety-critical (avionics, medical) → DO-178C, MC/DC, formal verification. E-commerce → functional, performance, security. Mobile app → usability, compatibility, network conditions. Internal CRUD tool → basic happy path and negative tests.
7
Absence-of-Errors Fallacy
Fixing all known defects does NOT guarantee a successful product. If the system does not meet user needs, it is still a failure. Links to Validation. Example: Microsoft Zune (2006) — worked as specified with no known defects, but failed commercially because users preferred the iPod ecosystem. Defect-free ≠ successful.

Section 4 — Software Testing Lifecycle (STLC)

Definition: A systematic process for planning, executing, and closing testing activities. Each phase has Entry Criteria, Activities, and Exit Criteria. Runs in parallel with the SDLC.

Phase 1 — Requirement Analysis

Identify testable vs. non-testable requirements ("The system shall be fast" is non-testable without a specific metric)
Create the RTM (Requirement Traceability Matrix) — links every requirement to one or more test cases. Columns: Req ID | Req Description | Test Case ID(s) | Test Status
Identify types of tests needed: functional, non-functional, regression, performance, security
Prepare automation feasibility report — which tests are good candidates for automation?

Phase 2 — Test Planning (IEEE 829)

Define test strategy: what types of testing and why?
Estimate testing effort; assign roles (Test Manager, Lead Tester, Automation Engineer, Performance Tester)
Select tools: defect tracking (Jira), test management (TestRail), automation (Selenium, Cypress)
Define entry and exit criteria for each phase; perform risk analysis

Phase 3 — Test Case Design

A good test case is: Clear, Traceable (linked to RTM), Independent, Repeatable.

Standard Test Case Template

Phase 4 — Environment Setup

Test environment must mirror production as closely as possible. Differences are a leading cause of "works in QA, breaks in production." Run a smoke test to verify stability before full execution.

Phase 5 — Test Execution

Severity vs. Priority — A Common Confusion

Severity = how much the defect impacts functionality. Priority = how urgently it must be fixed.

High Severity, Low Priority: App crashes when user enters a 300-character name — very rare edge case.

Low Severity, High Priority: Company logo displayed incorrectly on login page — brand issue, fix immediately.

Phase 6 — Test Closure

Key Metric — Defect Removal Efficiency (DRE)

DRE = (Defects found before release) ÷ (Defects found before + after release) × 100%

World-class: 95%+ | Industry average: ~85% | NASA target: 99%+

Section 6 — Cost of Quality (CoQ)

CoQ = Cost of Conformance + Cost of Non-Conformance

Introduced by Feigenbaum (1956); popularised by Crosby (1979): "Quality is free. What costs money are the unquality things."

Cost of Conformance (Good)

Prevention Costs: Training, process improvement, requirements reviews, design reviews, standards creation

Appraisal Costs: Testing (all types), inspections, audits, test environment setup, test tool licences

Cost of Non-Conformance (Bad)

Internal Failure: Rework, debugging, re-testing before delivery, schedule delays

External Failure: Production bug fixes, customer support, recalls, legal liability, reputation damage, lost customers

Activity 1 — Testing vs. Debugging Identification

Classify each scenario as Testing, Debugging, or Both:

#ScenarioClassificationJustification
1A developer runs all unit tests after changing a function.BothRunning the unit tests = Testing (systematic execution to detect failures). Changing the function in response to a failure = Debugging. Both activities occur together in TDD-style workflow.
2A QA engineer writes test cases for a new payment module.TestingWriting test cases is a Testing activity (specifically, the Test Case Design phase of STLC). No code is being fixed — the goal is to plan how to detect defects.
3A developer uses a step-through debugger to find why a variable is null.DebuggingUsing a debugger to locate the root cause of a null variable is pure Debugging — constructive activity aimed at finding and fixing the root cause, not detecting failures.
4A tester finds a crash and logs a defect report.TestingFinding the failure and logging a defect report is Testing — destructive in nature, aimed at detecting that the software has a failure. The actual fixing is left to the developer (Debugging).

Why keep testers and developers separate in formal testing?

Psychological bias: developers are naturally inclined to prove their code works (constructive mindset), whereas testers are trained to find failures (destructive mindset). Separate teams eliminate the "author's blind spot." Also, independent testing provides objective evidence of quality — a developer testing their own code will unconsciously avoid inputs they know are problematic.

Activity 5 — Challenge Root Cause Analysis

Example used: Knight Capital Group Algorithmic Trading Failure (2012)

QuestionAnswer
1. What was the FAILURE?The trading system executed 4 million unintended trades in 45 minutes, causing a $440 million loss and nearly bankrupting the company. Observable incorrect behaviour: trades were placed at the wrong prices and volumes, opposite to intended strategy.
2. What was the FAULT?A dormant, deprecated code path (old "Power Peg" functionality) was accidentally re-activated during deployment. The deployment script was applied to only 7 of 8 servers — one server still ran the old code. The fault was the active deprecated code remaining in the production codebase without proper guards or removal.
3. What human ERROR caused it?A developer reused a previously used feature flag to activate new functionality — not realising the same flag activated the dormant old code on servers that had not received the new deployment. This is a configuration management error combined with inadequate code cleanup (dead code not removed).
4. Which testing challenges contributed?Challenge 3 (Lack of Testability): the legacy code path had no safeguards or kill switches. Challenge 5 (Defect Reproduction Difficulty): the bug only manifested with a specific deployment configuration across 8 servers. Challenge 6 (Inadequate Test Environment): the staging environment did not replicate the exact multi-server production deployment scenario.
5. Which STLC phase should have caught it?Phase 4 (Environment Setup) and Phase 5 (Test Execution). A production-identical environment with all 8 servers would have revealed the partial deployment issue. A pre-deployment regression test of the full system with production-identical configuration would have triggered the dormant code path. The missing element was a deployment verification test that confirmed ALL servers received the same code version before live trading began.

CLO-2 — Understand

Q6 — Explain
Explain the difference between Verification and Validation. Why are both required throughout the SDLC?
Verification asks "Are we building the product right?" — it is a static process that checks artefacts (requirements, design, code) against specifications without executing the software. Techniques include reviews, walkthroughs, inspections, and static analysis.

Validation asks "Are we building the right product?" — it is a dynamic process that executes the software and checks whether it meets actual user needs. Techniques include functional testing, UAT, and system testing.

Both are required because: Verification alone can produce a technically correct system that solves the wrong problem (Mars Climate Orbiter — units matched specification but not intent). Validation alone without verification misses structural defects in design and code. Together they provide complete assurance: the system is built correctly AND it meets user needs.
Q7 — Understand
Explain the Pesticide Paradox with a practical example. What are two solutions?
The Pesticide Paradox (Boris Beizer, 1990) states that if the same set of tests is run repeatedly, they will eventually stop finding new defects — just as insects develop resistance to the same pesticide.

Cause: The tests were designed for specific code paths. As software evolves, new paths are added that the old tests do not cover. The tests become stale.

Practical example: A regression suite written in 2022 covers all features from that time. A major architectural change in 2024 adds new code paths. The old suite runs green but does not exercise the new paths — giving false confidence.

Solutions: (1) Regularly review and update the test suite — add new test cases whenever new features or changes are made. (2) Use mutation testing — deliberately introduce small code mutations to verify that tests actually detect the change (if the suite doesn't catch mutations, it is ineffective).
Q8 — Understand
Explain why Principle 3 (Early Testing Saves Money) is important for project management. Use Boehm's cost multipliers to support your answer.
Principle 3 (Shift-Left Testing) states that the cost of fixing a defect increases dramatically the later it is found in the SDLC. Boehm's (1981) cost multipliers show:

Requirements: ×1 (baseline) — change a sentence in a document
Design: ×5 — redesign one or more modules
Coding: ×10 — refactor already-written code
Unit Testing: ×20 — fix code and re-run affected tests
System Testing: ×40 — fix plus full regression of the entire system
Production: ×100+ — hotfix, customer impact, reputation damage, legal risk

Project management implication: Investing in early activities (requirements reviews, design inspections, unit testing) is far cheaper than finding and fixing defects late. The STLC starts at Phase 1 (Requirement Analysis) — not after development — specifically because of this cost curve. A team that skips design reviews to save time will spend far more time fixing system-level defects later.
Q9 — Understand
Distinguish between defect Severity and Priority. Give one example of each combination: (a) High Severity, Low Priority and (b) Low Severity, High Priority.
Severity measures how much the defect impacts the system's functionality (technical impact). Priority measures how urgently the defect must be fixed (business urgency).

(a) High Severity, Low Priority: The application crashes when a user enters a name longer than 300 characters. This is a serious functional failure (high severity) but the scenario is extremely rare — almost no user would type that much. The business impact is low, so it can be scheduled for a future release (low priority).

(b) Low Severity, High Priority: The company logo is displayed at the wrong size on the login page. Functionally, the app works perfectly — no feature is broken (low severity). However, this is the first screen every user sees and damages brand perception — it must be fixed immediately (high priority).

Scenario-Based Questions

Scenario Q1
A fintech startup builds a loan calculation engine. All unit and integration tests pass. Three weeks after launch, customers report that loans in a specific currency (JPY) are calculated incorrectly — the system uses the wrong decimal placement. (a) Identify this as Error, Fault, or Failure at each stage. (b) Which testing principle does this illustrate? (c) Which testing challenge contributed to this not being caught?
(a) Error: A developer assumed all currencies use 2 decimal places — not knowing JPY uses 0 decimal places (¥100 not ¥1.00). This is a human mistake during design/coding. Fault: The incorrect decimal handling code exists in the loan calculation engine for JPY inputs. Failure: Customers in Japan receive incorrect loan amounts displayed (e.g., ¥50,000 displayed as ¥500.00) — observable incorrect behaviour in production.

(b) Principle 1 — Testing Shows the Presence of Defects. Passing all unit and integration tests did not prove the software was defect-free. The specific JPY currency edge case was not covered by the test suite, so the underlying fault was never activated in testing.

(c) Challenge 1 — Incomplete or Vague Requirements: the requirement did not specify currency-specific decimal handling rules. If acceptance criteria had explicitly listed all supported currencies and their formatting rules, test cases would have covered JPY. Also possibly Challenge 2 (Insufficient Time/Budget) if edge-case currencies were deprioritised to meet a deadline.
Scenario Q2
A QA manager presents two test environment options to the project board: Option A — a minimal shared staging server that differs from production in OS version and database size. Option B — a full containerised environment using Docker that mirrors production exactly, but costs more to set up. The board asks the manager to justify Option B using Cost of Quality principles.
The manager should argue using CoQ framework:

Option A's hidden costs (Non-Conformance): Gartner estimates ~30% of production incidents are caused by environment differences. A defect that only manifests in production due to the different OS version or database size will be discovered after release — costing ×100 compared to finding it in testing. Additionally, debugging environment-specific issues wastes developer time (internal failure cost) and erodes team confidence in QA results.

Option B's investment (Conformance — Appraisal Cost): Setting up Docker containers is a one-time cost. Once written, the environment can be reproduced identically on every test run at near-zero marginal cost. It eliminates the "works in QA, breaks in production" problem, reduces re-work, and increases the reliability of test results as evidence of production quality.

Conclusion: The upfront investment of Option B prevents far more expensive external failure costs. This is the core principle of Cost of Quality — conformance spending prevents non-conformance spending, which is always higher.
Scenario Q3
A healthcare application development team has been using the same 800-test regression suite for 2 years. A new doctor-facing module is added. The test manager says "regression suite is green, ship it." A junior tester raises a concern. What principle supports the tester's concern, and what should the team do?
The junior tester's concern is supported by two principles:

Principle 5 — Pesticide Paradox: The 2-year-old regression suite was written to test the old system. The new doctor-facing module introduces new code paths, workflows, and data interactions that the existing 800 tests do not cover. The suite being "green" only means the old functionality still works — it says nothing about the new module.

Principle 1 — Testing Shows Presence of Defects: A green regression result does not prove the new module is defect-free. It proves no defects were found by the old tests — which were never designed to test the new module.

What the team should do: (1) Write new test cases specifically for the doctor-facing module covering all new user stories and acceptance criteria. (2) Update the RTM to link new requirements to new test cases. (3) Given this is a healthcare application (safety-critical context — Principle 6), apply more thorough testing with higher coverage requirements. (4) Run the new test cases alongside the existing regression suite before release.
Scenario Q4
An e-commerce company's QA team completed all planned testing and found no defects. The product owner declares "the system is defect-free — we can ship with full confidence." As a software quality engineer, how would you respond, and which testing principles support your position?
The response should challenge the "defect-free" claim using two principles:

Principle 1 — Testing Shows Presence of Defects (NOT their absence): Finding no defects in testing means the tests did not reveal any defects — not that none exist. Undiscovered defects may remain. The correct statement is "no defects were found by our test suite" — not "the system is defect-free." (Dijkstra, 1972)

Principle 2 — Exhaustive Testing is Impossible: The team could not have tested every possible input combination, user path, and environmental condition. Some defects may only appear under specific combinations not covered by the test suite.

Additionally, even if the system truly has no defects: Principle 7 — Absence-of-Errors Fallacy warns that a defect-free system can still fail commercially if it does not meet user needs or is unusable (Validation must also be confirmed).

Recommended response: "We can say that all planned tests passed and no defects were found within the scope of our testing. We cannot guarantee the system is defect-free. Our DRE and coverage metrics tell us how much confidence we have, but residual risk always exists. The decision to ship is a business risk decision, not a technical certainty."

Verification → "right?" static, specs Validation → "right product?" dynamic, users Reviews, walkthroughs = Verification UAT, system testing = Validation ESLint, SonarQube = Verification Functional testing = Validation

STLC Phases — One-liner Each

1
Requirement Analysis — Identify testable requirements, build RTM
2
Test Planning — Write master Test Plan (IEEE 829), assign roles, select tools
3
Test Case Design — Write clear, traceable, independent, repeatable test cases
4
Environment Setup — Mirror production; run smoke test before execution
5
Test Execution — Execute, log defects, re-test fixes, run regression
6
Test Closure — Test summary report, DRE metric, lessons learned, archive artefacts
