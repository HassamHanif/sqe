Section 1 — Testing Scopes & the V-Model

What are Testing Scopes? They define the boundary and objective of a test effort — what part of the system is being tested, at what level, and under what conditions. There are four standard scopes forming a progression from smallest to largest.

The V-Model (Verification & Validation)

Each development phase on the LEFT arm maps directly to a test phase on the RIGHT arm. Tests are planned early during design, and executed later as the system is built.

Section 2 — Unit Testing

Definition: Testing the smallest independently testable parts (functions, methods, classes) in complete isolation from the rest of the system.

Goals of Unit Testing

Correctness — verify each unit behaves per its specification
Regression protection — alert when a change breaks existing functionality
Safe refactoring — failures become immediately visible
Live documentation — executable specification of intended behaviour

Key Characteristics

CharacteristicDetail
ScopeSingle function, method, or class
IsolationNo database, no network, no file system — all dependencies replaced
SpeedMilliseconds per test; thousands can run in seconds
AuthorWritten by the developer
FrameworksJUnit 5 (Java), pytest (Python), NUnit/xUnit (.NET), Jest (JS)
Run frequencyEvery commit (CI/CD) or every save (TDD)

Test Doubles — The 5 Types

Because unit tests run in isolation, all external dependencies are replaced with Test Doubles:

TypeDefinitionExample
StubReturns pre-configured fixed values. No logic.Hard-coded list instead of real DB query
MockRecords calls and allows assertions on how it was called.Verify email service called once with correct params
FakeWorking simplified implementation, not for production.In-memory HashMap instead of real DB
SpyWraps real object, records calls transparently.Wrap logger to verify it was called, but still log
DummyPassed as required parameter but never used.Placeholder User object when only ID is needed

The AAA Pattern (Arrange–Act–Assert)

Every well-written unit test follows AAA — it makes tests self-documenting.

PhasePurposeWhat to do
ArrangeSet up preconditionsCreate objects, configure stubs/mocks, prepare input data
ActExecute the behaviour under testCall the single function being tested
AssertVerify the outcomeCheck actual result equals expected; one assertion preferred

AAA Example (Python / pytest)

def test_fine_for_5_days():
    # Arrange
    days = 5
    # Act
    result = calculate_fine(days)
    # Assert
    assert result == 2.50  # 5 * 0.50

def test_negative_days_raises_ValueError():
    with pytest.raises(ValueError):
        calculate_fine(-1)

Section 3 — Integration Testing

Definition: Verifies that two or more modules work correctly when combined. Confirms that interfaces and data flows between pieces are correct.

Common Defects Caught at Integration Level

Mismatched data types between modules Wrong API contracts Timing/sync errors Database interaction bugs Third-party service failures

Integration Strategies

StrategyApproachProsConsBest For
Big BangAll modules integrated at onceNo stubs/drivers neededHard to isolate failures; high riskSmall simple systems
Top-DownStart from top-level; stubs for lower modulesArchitecture validated earlyMany stubs; lower modules tested lateArchitecture is the main risk
Bottom-UpStart from lowest level; drivers for upper modulesLow-level modules tested thoroughlyUser-visible behaviour tested lastSystems built on critical utilities
Sandwich (Hybrid)Top-Down + Bottom-Up simultaneouslyReduces stubs and driversComplex to manageLarge layered systems

Key Concept — Stubs vs Drivers:

Stub = a fake lower-level module used in Top-Down testing

Driver = a fake upper-level module used in Bottom-Up testing

Section 5 — Testing Approaches

Black-Box Testing

Evaluates external behaviour without any knowledge of internal structure. Tester knows only inputs and expected outputs. Best for: functional testing, acceptance testing, regression testing.

TechniquePrincipleExample (Age: valid 18–65)
Equivalence Partitioning (EP)Divide input into classes expected to behave identically. Test one representative per class.Partitions: <18 invalid, 18–65 valid, >65 invalid. Test: 10, 30, 70
Boundary Value Analysis (BVA)Test at and around edges of partitions — where defects concentrate.Test: 17, 18, 19 (lower); 64, 65, 66 (upper)
Decision Table TestingEnumerate all combinations of input conditions and map to expected actions.Login: valid user AND valid pass → grant. All 4 T/F combos mapped.
Use-Case / Scenario TestingTest complete end-to-end user workflows.Student: search → check availability → reserve → collect

White-Box Testing

Designs test cases based on internal logic and structure of code. Tester has full access to source code. Also called structural or glass-box testing. Best for: unit testing, security auditing, safety-critical systems.

Coverage TypeDefinitionIndustry Target
StatementEvery executable statement executed at least once70–80%
BranchEvery branch (true AND false) of every decision executed80–90%
ConditionEach boolean sub-condition evaluated to both true and false85–95%
PathEvery distinct execution path through the function exercised100% (often infeasible)
MC/DCEach condition independently affects the decision outcome100% required (aviation, railway, safety)

Branch Coverage Example:

def classify_score(score):
    if score >= 50:      # Branch True → test with 75
        return 'Pass'
    else:                # Branch False → test with 30
        return 'Fail'
# Two tests achieve 100% branch coverage: test_pass(75) and test_fail(30)

Grey-Box Testing

Combines both perspectives — tester has partial knowledge (API contracts, data models, architecture) but still approaches the system externally as a user. Best for: integration testing, API testing, penetration testing.

Techniques: Matrix Testing, Regression Testing, Pattern Testing, Orthogonal Array Testing.

Approach Selection Guide (Exam Favourite)

ScenarioApproachRationale
Testing a sorting algorithmWhite-BoxNeed to verify all code paths; branch/path coverage required
Testing a login formBlack-BoxTest valid/invalid combos without needing implementation details
Testing REST API endpointsGrey-BoxKnow API contract but not full implementation
100% coverage of safety moduleWhite-BoxMC/DC coverage mandated by safety standards
UAT before go-liveBlack-BoxUsers have no interest in code internals; behaviour only
Regression after a bug fixGrey-BoxCombine automated regression with targeted checks at fix site
Security penetration testGrey-BoxAttacker has partial knowledge; uses both external and structural techniques

In CI/CD pipelines: static linters run on every commit; dynamic test suites run on every build.

Activity 1 — Writing Your First Unit Test

Task:

Write at least 4 unit tests for calculate_discount(price, loyalty_years) using the AAA pattern.

Rules: 0–1 years = 0% | 2–4 years = 10% | 5+ years = 20% | ValueError if price < 0 or loyalty_years < 0

Solved — 4 Tests Using AAA Pattern

import pytest

# Test 1 — 0-1 years loyalty → 0% discount
def test_no_discount_for_new_customer():
    # Arrange
    price = 100
    loyalty_years = 1
    # Act
    result = calculate_discount(price, loyalty_years)
    # Assert
    assert result == 100.0  # 0% discount

# Test 2 — 2-4 years loyalty → 10% discount
def test_ten_percent_discount():
    # Arrange
    price = 200
    loyalty_years = 3
    # Act
    result = calculate_discount(price, loyalty_years)
    # Assert
    assert result == 180.0  # 200 - 10% = 180

# Test 3 — 5+ years loyalty → 20% discount
def test_twenty_percent_discount():
    # Arrange
    price = 500
    loyalty_years = 7
    # Act
    result = calculate_discount(price, loyalty_years)
    # Assert
    assert result == 400.0  # 500 - 20% = 400

# Test 4 — ValueError on negative price
def test_negative_price_raises_ValueError():
    # Arrange / Act / Assert combined using pytest.raises
    with pytest.raises(ValueError):
        calculate_discount(-50, 3)

# Bonus Test 5 — ValueError on negative loyalty_years
def test_negative_loyalty_years_raises_ValueError():
    with pytest.raises(ValueError):
        calculate_discount(100, -1)

Test Doubles Analysis

If calculate_discount called an external pricing API:

Stub — Replace the pricing API call with a hard-coded fixed price value (e.g., return 100 always) to keep the test isolated and fast.
Mock — Verify the API was called exactly once and with the correct product ID parameter.
No Fake, Spy, or Dummy needed here since the API is a single external dependency returning a simple value.

Activity 2 — Integration Strategy Selection

Scenario:

Three-tier web application — Presentation Layer (React frontend) | Business Logic Layer (Node.js API) | Data Layer (PostgreSQL DB)

Q1: Which integration strategy and why?

Recommended: Bottom-Up strategy.

The Data Layer (PostgreSQL) is the most critical shared dependency — all business logic and presentation depends on it. By testing the Data Layer first, then moving up to Business Logic, then Presentation, we ensure the foundation is solid before testing higher-level behaviour. Defects in the Data Layer are easier to localise when tested in isolation before being combined with other layers.

Alternatively, Sandwich (Hybrid) is also defensible: test the Data Layer bottom-up while the Presentation Layer is tested top-down, meeting in the middle at the Business Logic Layer.

Q2: Stubs or Drivers needed?

StrategyWhat you need
Bottom-UpTest Drivers — fake Node.js API calls to test the PostgreSQL DB layer in isolation. Fake React frontend to test the Node.js API layer in isolation.
Sandwich (if chosen)Stubs for the DB when testing top-down from Presentation; Drivers for Presentation when testing bottom-up from DB.

Q3: One specific integration defect between Business Logic and Data layers

Example defect: The Node.js API sends a user ID as a String in the SQL query, but the PostgreSQL schema defines the user_id column as an INTEGER. Each module passes unit tests in isolation (Node.js returns the string, PostgreSQL accepts integers) — but when combined, the type mismatch causes a query to return no results or throw a type error. This defect would ONLY be caught at integration level.

Activity 3 — Approach Matching (Banking App)

Banking Mobile Application — Select Black / White / Grey-box for each:

a. Login function

Approach: Black-Box

Reason: The tester inputs valid/invalid usernames and passwords and checks whether access is granted or denied. No need to see the internal authentication algorithm. Equivalence partitioning and boundary value analysis apply (valid credentials, invalid password, locked account, empty fields). The behaviour — not the code — matters here.

b. Interest calculation algorithm

Approach: White-Box

Reason: Interest calculations involve complex conditional logic (e.g., different rates for different account types, compound vs simple interest, penalty clauses). All branches and conditions must be verified to be correct — a developer must examine the code paths to ensure the formula is correctly implemented. Branch coverage and condition coverage are essential here.

c. UI transaction flow (transfer money screen)

Approach: Black-Box

Reason: UI testing validates that the user journey works end-to-end — enter amount, select recipient, confirm, receive success message. The tester acts as an end user with no knowledge of the internal code. Scenario/use-case testing is the primary technique. The internal implementation of the transfer is irrelevant at this level.

d. REST API between mobile app and backend

Approach: Grey-Box

Reason: The tester knows the API contract (endpoint URLs, request/response format, HTTP status codes, authentication headers) but does not have access to the full backend implementation. Tests are made with real HTTP calls, verifying that the API behaves according to its documented contract. This is classic grey-box — partial knowledge of the internal structure (schema, auth mechanism) combined with external black-box test execution.

CLO-1 — Remember (Recall & Distinguish)

Q1 — Definition
Define the four testing scopes and state the order in which they are typically executed in the V-Model.
Unit Testing → Integration Testing → System Testing → Acceptance Testing.

Unit: tests individual functions/methods in isolation by the developer.
Integration: tests interactions between 2+ modules.
System: tests the complete product against requirements in a production-like environment.
Acceptance: final testing by end users/stakeholders to confirm business needs are met.
Q2 — Recall
Name all five types of Test Doubles and state what makes each unique.
Stub — returns pre-configured fixed values, no logic.
Mock — records calls and allows assertions on how it was called.
Fake — a working simplified implementation not suitable for production (e.g., in-memory DB).
Spy — wraps a real object and records calls transparently.
Dummy — passed as a required parameter but never actually used or called.
Q3 — Recall
What do the letters AAA stand for in the AAA pattern? What happens in each phase?
Arrange: Set up preconditions — create objects, configure stubs/mocks, prepare input data.
Act: Execute the behaviour under test — call the single function or method being tested.
Assert: Verify the outcome — check that actual result equals expected result; one assertion preferred.
Q4 — Recall
List the four integration strategies. For each, state whether it uses stubs, drivers, or neither.
Big Bang — neither (all modules integrated at once).
Top-Down — uses Stubs (for lower-level modules not yet ready).
Bottom-Up — uses Drivers (for upper-level modules not yet ready).
Sandwich — uses both Stubs AND Drivers simultaneously.
Q5 — Recall
Name the four sub-types of Acceptance Testing and state who performs each.
UAT (User Acceptance Testing) — performed by end users/business stakeholders.
Alpha Testing — performed by the internal development team at the developer's site.
Beta Testing — performed by selected external users in a real environment.
Contract Acceptance Testing — performed by the customer/client team to verify contractual deliverables.

CLO-2 — Understand (Explain & Apply)

Q6 — Explain
Explain how the V-Model maps development phases to testing phases. Why is this mapping useful?
In the V-Model, each left-arm development phase has a corresponding right-arm test phase: Requirements Analysis ↔ Acceptance Testing; System Design ↔ System Testing; Architectural Design ↔ Integration Testing; Module Design ↔ Unit Testing.

Tests are planned early (during the design phase) and executed later (when the code is built). This is useful because: (1) testers understand requirements before implementation; (2) test cases are prepared in parallel with development, saving time; (3) defects in design can be caught before coding begins.
Q7 — Understand
What is the difference between Equivalence Partitioning and Boundary Value Analysis? When would you use each?
Equivalence Partitioning (EP): divides the input domain into classes expected to behave identically. Test one representative value per class to reduce total test cases while maintaining coverage. Use when there is a large input domain.

Boundary Value Analysis (BVA): tests at and around the edges (boundaries) of equivalence partitions — where defects concentrate. Complements EP. Use for all numeric ranges.

They work together: EP identifies the partitions; BVA then tests the boundaries of each partition.
Q8 — Understand
Why is Statement Coverage considered a weaker coverage criterion than Branch Coverage? Illustrate with an example.
Statement Coverage only requires each executable statement to be executed once. It does NOT require that both outcomes (true and false) of conditional statements are tested.

Example: if score >= 50: return 'Pass' — a single test with score=75 achieves 100% statement coverage if there is no else branch written. But if there is an else branch, statement coverage alone might miss it.

Branch Coverage requires both the true AND false branch of every decision to be executed. It subsumes statement coverage. Two tests — score=75 (pass) and score=30 (fail) — achieve 100% branch coverage, catching defects in both paths.
Q9 — Understand
Explain why Static Testing is considered more cost-effective than Dynamic Testing when applied early in the development lifecycle. Cite the relevant finding.
Static testing (reviews, inspections, static analysis) finds defects in artefacts (requirements, design, code) without executing the software. When a defect is found early — such as in the requirements specification — it can be fixed by changing a document before any code is written.

According to Boehm (1981), defects found at the requirements stage cost 10–100× less to fix than the same defects found in production. This is because production defects require: finding the root cause in code, fixing it, retesting, re-deploying, and potentially compensating affected users.

Additionally, Fagan Inspections have been shown to find 60–90% of defects before execution.

Scenario-Based Questions

These require applying knowledge to a new situation — a common exam format. Read carefully.

Scenario Q1
A developer writes a function calculate_tax(income, country). Unit tests pass. The function is then integrated with a CurrencyConverter module. A bug is discovered where the tax is calculated on the unconverted income (USD) instead of the converted local currency. (a) At which testing scope was this defect caught? (b) What type of test double would have been used in the unit test for calculate_tax, and why didn't it catch this bug?
(a) This defect was caught at Integration Testing scope. The unit tests for calculate_tax used an isolated test double for CurrencyConverter. Only when the real modules were combined did the incorrect data flow (wrong currency value being passed) become apparent.

(b) A Stub was used in the unit test — it returned a hard-coded converted value regardless of input. This meant the unit test of calculate_tax always received the "correct" currency format. The real CurrencyConverter was not exercised. Unit tests confirm individual module logic; they cannot detect interface mismatches between modules — that is precisely what integration testing is for.
Scenario Q2
You are testing a hospital patient management system. The medication dosage calculator must comply with IEC 61508 safety standards. (a) Which testing approach should you use? (b) Which coverage criterion is required? (c) Who should perform this testing?
(a) White-Box testing — because the internal logic of the dosage calculator must be fully verified. We cannot rely on black-box testing alone for safety-critical systems, as we need to confirm every code path computes the correct dosage.

(b) MC/DC (Modified Condition/Decision Coverage) at 100% is required. IEC 61508 (safety standard for functional safety in electrical/electronic systems) mandates MC/DC to ensure each condition independently affects the decision outcome — critical when wrong dosage can harm patients.

(c) The QA team or a specialist safety testing team performs this — not end users. They need code access and coverage tools.
Scenario Q3
A software company delivers an e-commerce platform to a client. Before final delivery, they perform multiple rounds of testing. (a) Library staff (client employees) test whether they can complete a full purchase workflow. (b) A group of 200 volunteer customers use the system for 2 weeks in a live environment. (c) The client's legal team verifies that all 23 features specified in the contract are implemented. Identify the acceptance testing sub-type for each.
(a) UAT (User Acceptance Testing) — performed by end users (client staff) testing real business workflows against their daily procedures.

(b) Beta Testing — performed by selected external users in a real environment to gather real-world feedback before the general release.

(c) Contract Acceptance Testing — performed by the customer/client team to verify that all contractual deliverables (the 23 listed features) have been implemented as agreed.
Scenario Q4
A QA team is testing a food delivery mobile app. For each task below, select the most appropriate testing approach (Black-Box, White-Box, or Grey-Box) and justify: (a) Verifying that the address validation rejects P.O. boxes. (b) Ensuring the route optimisation algorithm visits all delivery stops efficiently. (c) Testing the REST API that connects the app to the restaurant backend.
(a) Black-Box — the tester enters addresses (with and without P.O. box formats) and checks the output (accepted/rejected). No knowledge of validation logic internals is needed. Equivalence partitioning and BVA apply.

(b) White-Box — the route optimisation algorithm contains complex internal logic (graph traversal, cost functions, conditions). To verify correctness and efficiency, the tester must examine code paths and ensure all branches and conditions are exercised. Path and branch coverage are required.

(c) Grey-Box — the tester knows the API contract (endpoints, request/response format, authentication mechanism) but does not have access to the full backend implementation. Real HTTP calls are made with valid and invalid payloads, leveraging partial knowledge of the system architecture.
Scenario Q5
A development team uses a CI/CD pipeline. Explain how both Static and Dynamic testing would be integrated into this pipeline, and state what each type of testing would catch.
In a CI/CD pipeline, both types run automatically at different stages:

Static Testing — runs on every commit (before build): ESLint/Pylint check code style and undefined variables; SonarQube analyses code quality (dead code, duplication, complexity); SAST tools (Fortify/Veracode) scan for security vulnerabilities. These catch: code style violations, security bugs, unreachable code, logic errors — without executing the software.

Dynamic Testing — runs on every successful build: Unit tests (pytest/JUnit) run first; integration tests run after; E2E tests (Selenium/Cypress) run on the staging environment. These catch: runtime errors, incorrect outputs, performance bottlenecks, integration interface mismatches.

Together they implement the principle that static testing is most cost-effective early (preventing defects from propagating) while dynamic testing provides ground truth on actual runtime behaviour.

Quick Revision — All key facts for rapid review before exam

The 6 Key Takeaways

1
Scopes: Unit (isolation) → Integration (interactions) → System (end-to-end) → Acceptance (user needs). Each scope catches different defect types by different people.
2
Unit Testing: Smallest testable unit, in isolation, using Test Doubles (Stub/Mock/Fake/Spy/Dummy), following AAA pattern. Written by the developer.
3
Integration Testing: Verifies component interactions. Strategies: Big Bang (no stubs/drivers), Top-Down (stubs), Bottom-Up (drivers), Sandwich (both).
4
System & Acceptance: System = complete system vs requirements (functional + non-functional). Acceptance = UAT / Alpha / Beta / Contract — validates business needs.
5
Approaches: Black-box (behaviour/external) | White-box (code paths/coverage criteria) | Grey-box (partial knowledge/APIs). Choice depends on context and what you know.
6
Static vs Dynamic: Static = no execution (reviews, linters, SAST). Dynamic = execution required (all active testing). Both are complementary — mature process uses both.

Must-Know Facts (High Exam Probability)

Fagan Inspections — introduced by Michael Fagan at IBM in 1976. Finds 60–90% of defects before execution.
Boehm's Cost Curve (1981) — defects found at requirements stage cost 10–100× less to fix than production defects.
MC/DC Coverage — required at 100% by DO-178C (aviation), EN 50128 (railway), IEC 61508 (safety systems).
Branch Coverage subsumes Statement Coverage — if you have 100% branch, you automatically have 100% statement. Reverse is NOT true.
Unit tests don't catch integration defects — a type mismatch between two modules (e.g., String vs Integer) can only be caught at Integration Testing level.
System Testing is black-box — internal design is NOT the focus; only observable behaviour is tested.
Acceptance Testing is lowest automation — mostly manual because real users are involved and business processes must be validated.
