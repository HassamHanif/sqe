What is Software Quality?

Slide 3 — Section 1
Official ISO/IEC 25010:2011 Definition

"The degree to which a software product satisfies stated and implied needs when used under specified conditions."

— ISO/IEC 25010:2011

This definition has four key components — all are important for exam:

📋 Definition Breakdown

  • Stated requirements — explicitly documented requirements (e.g., login must load in 2 sec)
  • Implied expectations — unstated but expected (e.g., app shouldn't crash)
  • Standards conformance — follows accepted coding/design standards
  • Fitness for purpose — works correctly in the real-world context it was built for

✅ Why Quality Matters

  • Cost reduction — fixing a bug early is 10× cheaper than fixing it after release
  • User trust — poor quality leads to negative reviews and user abandonment
  • Maintainability — quality code is easier to extend and modify over time
  • Safety-critical systems — low quality can cost lives (medical, aerospace, finance)
💡 Exam Tip: The ISO definition says "degree" — meaning quality is measurable on a scale, not binary (good/bad). Also remember: stated = written down in requirements; implied = assumed but not written.

Quality in Context — 4 Stakeholder Perspectives

Slide 4 — Section 1

Different stakeholders evaluate quality from different angles. All perspectives are equally valid:

👤 User's View

Does it do what I need? Is it easy and pleasant to use? Focuses on usability and functionality.

💻 Developer's View

Does code meet specifications? Is it well-structured and testable? Focuses on code quality and design.

📊 Manager's View

Delivered on time, within budget? Minimizes rework? Focuses on process efficiency and cost.

🔍 Tester's View

How many defects remain? Are requirements fully covered? Focuses on defect detection and coverage.

⚠️ Exam Tip: In a scenario question, identify which stakeholder is speaking and what their quality concern is. E.g., "The app keeps crashing during payment" → User's View (reliability/usability).

Software Quality Attributes

Slide 5 — Section 2

Functional Attributes

WHAT the software does

  • Correctness — software does exactly what it is supposed to do according to specs
  • Completeness — all required features are present and implemented
  • Consistency — behaves predictably across different contexts and conditions
  • Traceability — every requirement is linked to an implementation element

Non-Functional Attributes

HOW WELL the software performs

  • Performance — speed, throughput, response time
  • Reliability — consistent correct operation over time without failure
  • Usability — ease of learning and day-to-day use
  • Maintainability — ease of modification, repair, and update
  • Portability — ability to run across different environments/OS/hardware
  • Security — protection against unauthorized access or data breach
💡 Memory Trick: Functional = what it does (like a job description). Non-functional = how it does it (like performance standards). Functional has 4 attributes; Non-functional has 6 key ones.

Introduction to Quality Engineering (QE)

Slides 8 & 9 — Section 4
Definition of Quality Engineering

The application of engineering principles and practices to ensure that software products meet defined quality standards throughout the development lifecycle.

QE operates on 4 core activities (the four pillars):

🛡️
Prevention
Build quality in from the start — requirements, design, code reviews
🔍
Detection
Find defects early via inspections, testing, and static analysis
🔧
Correction
Fix defects and analyse root causes to prevent recurrence
📊
Process Control
Monitor and measure quality throughout the SDLC

What Quality Engineers Do (6 key activities):

📋 Requirements Review

Analyse and validate requirements for completeness, consistency, and testability before development begins.

🧪 Test Planning & Design

Define test strategies, write test plans, design test cases aligned with requirements and acceptance criteria.

🔎 Inspections & Walkthroughs

Structured reviews of code, design documents, and specifications to detect defects early.

📈 Metrics & Measurement

Collect and analyse quality metrics: defect density, coverage, reliability — to track and improve quality.

🔄 Process Improvement

Identify root causes of quality issues and improve development and testing processes continuously.

⚙️ Automation & Tooling

Build automated test suites and integrate quality gates into CI/CD pipelines for continuous assurance.

💡 Key Point: QE is not the same as testing. Testing is one activity within QE. QE spans the entire SDLC — from requirements review to post-release process improvement.

Quality vs. Reliability

Slides 10–13 — Section 5
Software Quality Software Reliability
Definition Degree of conformance to requirements and fitness for purpose Probability of failure-free operation for a specified time/usage
Scope Broad — covers all 8 ISO 25010 characteristics Narrow — only ONE specific quality attribute (characteristic #5)
Measurement Qualitative + quantitative metrics (reviews, test coverage, security scans) Purely quantitative: MTTF, MTBF, failure rate
Standard ISO/IEC 25010 IEEE 982.1, MIL-HDBK-217
When assessed Throughout SDLC and after delivery During operation over time (measured in the field)
Examples Usability tests, code reviews, security scans MTTF calculations, reliability growth models
The Most Important Concept — Reliability ⊂ Quality

Reliability is a SUBSET of Quality. Quality contains reliability, not the other way around. High reliability does NOT automatically mean high overall quality.

Real-World Banking App Case Study (Slide 13):

✅ Ideal — Scenario A
High Quality + High Reliability

App meets all user needs, works consistently 99.99% of the time, is secure, fast, and easy to use. This is the target for every software product.

⚠️ Acceptable but Incomplete — Scenario B
High Reliability + Low Quality

App never crashes but the interface is confusing, transactions are slow, and features are missing. Reliable ≠ Quality. User will still be unhappy.

❌ Unacceptable — Scenario C
Low Reliability + Low Quality

App crashes frequently, has security vulnerabilities, and fails to complete transactions reliably. Completely unacceptable — especially in a safety-critical banking context.

💡 Key Formula to Remember: Reliability ⊂ Quality  |  Quality ⊃ Reliability  |  High Reliability ≠ High Quality

Activity 1

Identifying Quality Perspectives

Q1: A customer complains: "The app is too slow to load my account balance, and I can never find the transfer button." Which stakeholder perspective does this represent? Which quality attributes are being questioned?
✅ Model Answer

Perspective: User's View — the customer is describing their own experience of using the software.

Quality Attributes Questioned:
— "Too slow to load" → Performance Efficiency (time behaviour sub-characteristic)
— "Can never find the transfer button" → Usability (operability / recognisability sub-characteristic)

Q2: A project manager says: "We delivered this feature 3 weeks late and had to re-do the login module twice due to defects." Which perspective and quality concern does this reflect?
✅ Model Answer

Perspective: Manager's View

Quality Concern: Late delivery = failure of process quality control. Re-doing the login module = defect correction cost, which increases project cost and reduces efficiency. This relates to poor Prevention and Detection in the QE process — defects should have been caught earlier.

Q3: A developer says: "The module I was given has no documentation, functions are 500 lines long, and there's no test coverage." What quality attributes are affected?
✅ Model Answer

Perspective: Developer's View

Quality Attributes Affected:
— No documentation + 500-line functions → Maintainability (analysability, modifiability sub-characteristics)
— No test coverage → Maintainability (testability sub-characteristic)
— Also reflects poor functional suitability if the specs were not met properly.

Activity 2

ISO 25010 — Classify the Scenario

Q4: An e-commerce website works perfectly on Chrome but fails completely on Firefox. Which ISO 25010 characteristic does this violate?
✅ Model Answer

Characteristic: Compatibility (#3) — specifically the interoperability sub-characteristic.

The software is unable to interact and operate correctly across different browser environments. This may also touch Portability (#8) — specifically adaptability.

Q5: A hospital system stores patient passwords in plain text and was hacked, exposing all patient records. Which ISO 25010 characteristic is violated? Name two specific sub-characteristics.
✅ Model Answer

Characteristic: Security (#6)

Sub-characteristics violated:
— Confidentiality — patient data was exposed to unauthorized parties
— Integrity — data may have been altered/stolen, compromising its accuracy
— Also arguably Authenticity — authentication/password storage was inadequate

Q6: After a server crash, the online banking system restores full operation within 30 seconds without any data loss. Which ISO 25010 sub-characteristics are demonstrated? Name at least two.
✅ Model Answer

Characteristic: Reliability (#5)

Sub-characteristics demonstrated:
— Recoverability — the system recovered successfully after a failure (30 seconds)
— Fault Tolerance — the system continued without data loss despite the crash
— Availability — the system quickly became available again after the crash

Activity 3

Quality vs. Reliability — Distinction

Q7: "A military drone control software has never crashed in 10,000 flight hours, but pilots find the interface confusing and sometimes press the wrong buttons." Is this software high quality? Explain using the Quality vs. Reliability distinction.
✅ Model Answer

Reliability: HIGH — the software has operated failure-free for 10,000 flight hours (measured by MTTF). This means it scores highly on the Reliability characteristic of ISO 25010.

Overall Quality: LOW — The confusing interface that causes pilots to press wrong buttons indicates a severe failure in Usability (operability sub-characteristic). In a safety-critical system, poor usability can lead to catastrophic human error.

Conclusion: This confirms that Reliability ⊂ Quality. High reliability does NOT equal high overall quality. The software passes one of the 8 ISO characteristics but fails another critical one.

Q8: List three differences between Software Quality and Software Reliability using the comparison table from the lecture.
✅ Model Answer

1. Scope: Quality is broad (all 8 ISO 25010 characteristics); Reliability is narrow (only one specific characteristic).

2. Measurement: Quality uses both qualitative and quantitative metrics (reviews, test coverage); Reliability uses purely quantitative models (MTTF, MTBF, failure rate).

3. Standard: Quality is governed by ISO/IEC 25010; Reliability is governed by IEEE 982.1 and MIL-HDBK-217.

Bonus: Quality is assessed throughout the SDLC; Reliability is measured during operational use over time.

Activity 4

Quality Engineering — Applying the 4 Pillars

Q9: A development team is building a food delivery app. For each QE pillar (Prevention, Detection, Correction, Process Control), give one concrete example of what a Quality Engineer would do.
✅ Model Answer

Prevention: Conduct requirements review sessions before development begins to ensure the delivery tracking feature requirements are complete, consistent, and testable.

Detection: Perform code inspections and run automated unit tests during development to find defects in the order processing module early.

Correction: When a defect is found (e.g., incorrect price calculation), fix it, document the root cause (a wrong formula), and add a regression test to prevent recurrence.

Process Control: Measure defect density at each sprint, track test coverage percentage, and report to the team weekly so they can see if quality is improving or declining.

🔬 Section B — Application Questions (CLO-1 + CLO-2)

7 Distinguish between software quality and software reliability. Why is it incorrect to say "if a system is reliable, it must be high quality"? Apply ▼ Show Answer
5 MarksCLO-1

Software Quality is a broad, multi-dimensional concept that covers ALL 8 ISO 25010 characteristics — including usability, security, performance, maintainability, and reliability. It is assessed using both qualitative and quantitative methods throughout the SDLC.

Software Reliability is one specific characteristic within software quality — the probability of failure-free operation for a specified time/usage period. It is measured quantitatively using metrics like MTTF (Mean Time to Failure) and MTBF.

Why the statement is incorrect: Reliability is a SUBSET of quality (Reliability ⊂ Quality). A system can achieve high reliability (e.g., never crashes) while still scoring very poorly on other quality characteristics — for example, it may have a terrible user interface (low usability), be vulnerable to cyberattacks (low security), or be impossible to update without breaking (low maintainability). Therefore, high reliability alone does not guarantee high overall quality.

8 Compare the four stakeholder perspectives on software quality. Which perspective focuses most on the internal structure of the software? Apply ▼ Show Answer
4 MarksCLO-1

The four perspectives are:
— User's View: Usability and functionality — does it do what I need, is it easy to use?
— Developer's View: Code structure, specifications, and testability
— Manager's View: On-time delivery, budget adherence, and minimal rework
— Tester's View: Defect count, requirements coverage, and test completeness

The Developer's View focuses most on the internal structure of the software, as developers are directly concerned with code quality, architectural design, and whether the implementation is well-structured and follows specifications.

🎭 Section C — Scenario-Based Questions

9 SCENARIO: A ride-sharing app has been running for 2 years with 99.95% uptime and zero crashes. However, users complain that: (a) the app takes 8 seconds to show nearby drivers, (b) the payment screen is confusing, causing accidental double-charges, and (c) the app cannot be updated without rewriting 60% of the codebase. Identify which ISO 25010 characteristics are violated for each complaint and explain. Scenario ▼ Show Answer
9 Marks — 3 per pointCLO-1

(a) "8 seconds to show nearby drivers"
ISO 25010 Characteristic violated: Performance Efficiency (#2)
Sub-characteristic: Time behaviour — the system fails to meet acceptable response time requirements. 8 seconds for a core function (showing drivers) is unacceptable and indicates inefficient algorithms or resource utilization.

(b) "Confusing payment screen causing double-charges"
ISO 25010 Characteristic violated: Usability (#4)
Sub-characteristics: Operability (users cannot operate the payment function correctly) and User error protection (the system does not prevent user input errors from causing financial harm). Also potentially Functional Suitability (#1) if the double-charge is a functional defect.

(c) "Cannot update without rewriting 60% of codebase"
ISO 25010 Characteristic violated: Maintainability (#7)
Sub-characteristics: Modifiability (changes cannot be made without major impact), Modularity (the system is not designed in independent modules), and potentially Analysability (hard to understand what needs to change).

Note: The 99.95% uptime shows HIGH reliability, but these violations prove that Reliability ⊂ Quality — high reliability does not mean high overall quality.

10 SCENARIO: A software company has a new project to build an online exam system for a university. As a Quality Engineer, describe what you would do at each of the four QE pillars (Prevention, Detection, Correction, Process Control) during the project. Be specific to the exam system context. Scenario ▼ Show Answer
8 Marks — 2 per pillarCLO-1

Prevention: Before any coding begins, conduct a requirements review to ensure all exam system requirements are complete (e.g., does the system need to support 500 simultaneous users? Is auto-save specified?). Also review the system architecture design to identify any potential issues with scalability or security of student data before they become expensive code-level defects.

Detection: Perform code walkthroughs and inspections during development to find defects in the timer module, question randomization logic, and submission handling. Execute test cases including: timing edge cases (what happens at exactly 0 seconds left?), browser compatibility tests, and load testing with 500+ simultaneous users.

Correction: When defects are found (e.g., exam timer continues after submission, allowing cheating), fix them immediately, perform root cause analysis (e.g., the timer and submission events were not properly synchronized), and add a specific regression test to prevent the same defect recurring in future updates.

Process Control: At the end of each sprint, measure defect density (defects per 1000 lines of code), test case pass rate, and requirement coverage percentage. Report these metrics to the project manager and team so that quality trends are visible and the team can adjust their practices if quality is declining.

11 SCENARIO: Two students debate. Student A says "Quality and reliability mean the same thing — a reliable system is a quality system." Student B disagrees. Who is correct? Write a 6-line argument to settle the debate, using the ISO 25010 standard as evidence. Scenario ▼ Show Answer
5 MarksCLO-1

Student B is correct.

According to ISO/IEC 25010, software quality is a broad concept encompassing 8 distinct characteristics: Functional Suitability, Performance Efficiency, Compatibility, Usability, Reliability, Security, Maintainability, and Portability.

Reliability is just ONE of these 8 characteristics — it represents the probability of failure-free operation over time. A system that never crashes (high reliability) may still have a confusing interface (low usability), be vulnerable to hackers (low security), or be impossible to maintain (low maintainability).

Therefore, reliability is a subset of quality (Reliability ⊂ Quality), meaning high reliability is necessary but NOT sufficient for high overall quality. Student A's claim is incorrect because it treats one component as equivalent to the entire concept.

Key Terms & Definitions

Software Quality
The degree to which a software product satisfies stated and implied needs when used under specified conditions. (ISO/IEC 25010:2011)
Stated Requirements
Explicitly documented requirements that the software must fulfill.
Implied Needs
Unstated but expected user expectations that a reasonable user would assume the software fulfills.
Functional Attributes
Quality attributes related to what the software does (correctness, completeness, consistency, traceability).
Non-Functional Attributes
Quality attributes related to how well the software performs (performance, reliability, usability, security, maintainability, portability).
ISO/IEC 25010
International standard (2011) that defines 8 main quality characteristics for software product quality.
Quality Engineering (QE)
Application of engineering principles to ensure software products meet quality standards throughout the entire development lifecycle.
Software Reliability
The probability of failure-free operation of a system for a specified time/usage period under specified conditions. One of the 8 ISO 25010 characteristics.
MTTF
Mean Time To Failure — a reliability metric measuring the average time before a system failure occurs.
MTBF
Mean Time Between Failures — reliability metric measuring average time between consecutive failures (includes repair time).
Defect Density
A quality metric: number of defects per 1000 lines of code (or per function point). Used in Process Control.
Fitness for Purpose
The degree to which a software product fulfills the specific goals and needs of its intended users in their real context of use.
Fault Tolerance
A reliability sub-characteristic — the system continues to operate correctly despite the presence of hardware or software faults.
Recoverability
A reliability sub-characteristic — the ability of a system to recover affected data and re-establish its desired state after a failure.
Traceability
A functional attribute — every requirement can be traced to its corresponding implementation element, and vice versa.
SDLC
Software Development Life Cycle — the structured process for planning, creating, testing, and deploying software. QE spans the entire SDLC.
💡 Mnemonic for 8 ISO Characteristics: "Fine Products Come Under Rigorous Standard Management Protocols"
F = Functional Suitability · P = Performance Efficiency · C = Compatibility · U = Usability · R = Reliability · S = Security · M = Maintainability · P = Portability
